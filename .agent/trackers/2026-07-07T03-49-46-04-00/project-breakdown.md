## Current project shape

`ZombieOrchard` is a standalone kit-composed orchard survival and economy game shell.

The repo is intentionally small and static:

- `index.html` mounts a canvas, UI root, and error panel.
- `src/boot.js` imports `src/start.js`.
- `src/start.js` creates the game runtime, world renderer, HTML interface renderer, frame loop, and `window.GameHost` diagnostics surface.
- `src/game.js` composes the runtime from reusable kits and the orchard preset.
- `src/kits/runtime.js` provides the minimal kit runtime.
- `src/kits/scoped-interface-domains.js` defines reusable scoped interface domains.
- `src/kits/composition.js` handles screen/domain transitions.
- `src/kits/game-domains.js` owns resource, pressure, orchard world, construction, roster, inventory, and active-session behavior.
- `src/presets/orchard-preset.js` supplies project-specific menu, action, resource, pressure, construction, roster, and inventory presets.
- `tests/smoke.mjs` validates the entry-to-active-session transition and apple generation.

## Interaction loop

### Current playable loop

```txt
load page
→ create kit runtime
→ install game-state kits
→ install scoped interface domain kits
→ install interface composition kit
→ render entry screen
→ player presses Play
→ transition to active-session
→ render orchard canvas and HUD
→ player collects nearby apples
→ resource ledger gains apples and money
→ pressure field rises slightly
→ player clears reachable pests when present
→ player can advance day/night phase
→ night phase spawns pests over time
→ pests chase the player
→ player condition falls if pests reach them
→ if condition reaches 0, session ends
→ interface composition sends player to outcome
```

### Economy loop currently implied

```txt
collect apples
→ gain money
→ spend money and wood on buildings
→ use buildings/roster/inventory as management surfaces
→ survive row pressure and curse escalation
→ repeat across days
```

### Target product loop

```txt
title
→ choose or create orchard run
→ inspect orchard rows
→ harvest apples and rare gold apples
→ sell or spend resources
→ build storage/defense/production structures
→ hire field hands
→ manage inventory/tools
→ survive night pest pressure
→ expand orchard economy
→ unlock codex knowledge
→ reach run outcome
```

The current code has the backbone for this loop, but most management surfaces are still shallow screens rather than fully simulated economy subsystems.

## Services that kits offer

### `kit-runtime`

Current services:

- Install kits into the runtime.
- Store domains by id.
- Route domain commands.
- Tick every domain with a clamped delta.
- Emit per-frame events.
- Produce full runtime snapshots.
- Subscribe listeners to snapshot changes.

Service surface:

```txt
createKitRuntime({ kits })
engine.addKit(kit)
engine.command(domainId, type, payload)
engine.tick(delta)
engine.snapshot()
engine.subscribe(listener)
ctx.emit(type, payload)
```

### `scoped-interface-domain-kit`

Current services:

- Create reusable interface domains from config.
- Hold title, description, selected index, fields, metadata, and actions.
- Select actions.
- Set fields.
- Activate selected or explicit actions.
- Emit selection/action events.
- Snapshot interface state.

Service surface:

```txt
createScopedInterfaceDomainKit(domainId, config)
createInterfaceDomainKits(config)
command("select", { index })
command("set-field", { key, value })
command("activate", { actionId })
snapshot()
```

### `interface-composition-kit`

Current services:

- Track active and previous interface domain.
- Transition between configured domains.
- Activate actions on the active domain.
- Run action commands against target domains.
- Auto-route to outcome when the active session ends.
- Snapshot active screen and active screen state.

Service surface:

```txt
createInterfaceCompositionKit({ initial, transitions })
command("transition", { to })
command("back", { to })
command("activate", { actionId })
tick()
snapshot()
```

### `resource-ledger-kit`

## Current gaps

- Movement exists in the domain command surface, but no keyboard/pointer input mapper currently sends move commands.
- The market/exchange screen has no real buy/sell service yet.
- Roster can hire workers, but no worker automation loop exists.
- Inventory can equip items, but tools do not affect harvest, combat, stamina, or economy yet.
- Construction adds buildings, but buildings do not alter capacity, defense, production, or worker behavior yet.
- Save/select surfaces are present, but save persistence is not implemented.
- Knowledge/codex is present as a screen, but not connected to unlocks or discoveries.
- `orchard-world` uses nondeterministic `Math.random`, which limits replay/smoke determinism.
- `active-session` mixes gameplay, combat, phase, player, pest, score, and action routing responsibilities.
- Canvas visuals are readable but still placeholder rectangles.

Day 1:

  pests enter rows

Day 2:
  buy better tool

Night 2:
  stronger pressure
  defense buildings matter
