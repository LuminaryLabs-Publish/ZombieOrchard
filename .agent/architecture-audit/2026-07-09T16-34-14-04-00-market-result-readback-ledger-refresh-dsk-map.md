# ZombieOrchard Architecture Audit: Market Result Readback Ledger Refresh

**Timestamp:** `2026-07-09T16-34-14-04-00`

## Architecture summary

`ZombieOrchard` is composed as a static browser host plus a compact kit runtime.

```txt
index.html
  -> src/boot.js
  -> src/start.js
  -> src/game.js
  -> src/kits/runtime.js
  -> source kits and interface kits
  -> renderers
```

`src/game.js` is the composition root. It installs runtime/game/interface kits and returns the engine.

## Runtime boundary

```txt
createKitRuntime
  owns:
    ctx
    domains map
    engine.command(domainId, type, payload)
    engine.tick(delta)
    engine.snapshot()
    subscription notification
```

The runtime is already viable because `engine.command()` returns `{ accepted, reason? }` command results and snapshots are domain keyed. Do not replace this runtime for the next slice.

## Implemented DSK/domain map

```txt
kit-runtime
  -> kit-runtime domain services

orchard-world-kit

  -> services: tree generation, apple generation, collectNear, reseed, world snapshot

construction-runtime-kit
  -> construction-runtime domain
  -> services: build catalog lookup, resource payment, built row append, build message, snapshot

roster-runtime-kit
  -> roster-runtime domain
  -> services: hire, money payment, actor append, roster message, snapshot

inventory-runtime-kit
  -> inventory-runtime domain
  -> services: equip, inventory snapshot

scoped-interface-domain-kit
  -> entry/session-select/run-setup/interrupt/construction/exchange/roster/inventory/knowledge/preferences/outcome domains
  -> services: actions, select, set-field, activate, screen snapshot

active-session-domain-kit

  -> services: route DOM clicks, render HUD, render generic interface panels
```

## Current architecture problem

`interface-composition-kit` dispatches nested `action.command` through `ctx.engine.command(...)` but discards the nested result. That means an Exchange/Market action can mutate another domain without producing a stable consumer-visible result row.

The `exchange` screen is generated by `scoped-interface-domain-kit`, but it currently functions as a generic descriptor screen and does not own a Market source catalog, price rows, capacity policy, transaction journal, or accepted/rejected command-result schema.

## Required next-cut architecture

```txt
market-action-catalog-kit
  -> owns stable Exchange action ids and labels

market-command-source-manifest-kit
  -> owns source rows: price, resource delta, inventory delta, capacity, expected mutation

market-command-envelope-kit
  -> normalizes UI action into replayable command envelope

market-source-snapshot-kit
  -> captures before/after resources, inventory, prices, capacity, active screen

market-preflight-kit
  -> owns stable accepted/rejected reasons

market-command-result-kit
  -> owns accepted/rejected result schema

resource-transaction-history-kit
  -> records accepted resource mutations

inventory-purchase-intake-kit
  -> records accepted inventory mutations

market-command-journal-kit
  -> records attempted Market envelopes

market-result-journal-kit
  -> records Market result rows

interface-nested-result-adapter-kit
  -> preserves nested result in interface-composition snapshot.lastResult

market-result-projection-kit
  -> converts source/result state into Exchange UI projection

market-render-readback-kit
  -> records visible rows, last result, rejection reason, and transaction counts

market-gamehost-diagnostics-kit
  -> exposes fixture-readable diagnostics through window.GameHost.getState()

market-fixture-replay-kit
  -> proves accepted sell, accepted buy, rejected insufficient-resource, rejected capacity, and no-mutation rejection rows
```
