# Raw public-host gameplay bypass loop

## Goal

Document how external host access can mutate gameplay outside the intended interface, route and simulation admission paths.

## Intended gameplay loop

```txt
rendered action
  -> delegated click binding
  -> engine.command(...)
  -> interface composition or active-session command
  -> runtime notify
  -> next RAF tick and render
```

## Public bypass loop

```txt
external script or diagnostic
  -> window.GameHost.engine
  -> domains[id]
  -> api, command or tick
  -> mutation without the intended public command boundary
```

Examples reachable through the current graph:

```txt
resource-ledger.api.add(...)
  -> grants resources directly
  -> no command result or subscriber notification

pressure-field.api.adjust(...)
  -> changes pressure directly
  -> no command result or route/lifecycle admission

orchard-world.api.collectNear(...)
  -> removes and reseeds apples directly
  -> bypasses active-session score, message and reward composition

active-session.command("move", ...)
  -> moves the player directly
  -> bypasses interface command policy and runtime notify

active-session.tick(dt)
  -> advances pests and damage independently
  -> pressure and other domains do not advance in the same step

engine.tick(dt)
  -> advances all domains beside the production RAF
  -> callback cadence and diagnostic activity change gameplay speed

engine.addKit(candidate)
  -> can overwrite an existing domain ID
  -> redirects future gameplay calls to a replacement authority
```

## Consequences

```txt
unjournaled resource changes
partial-domain simulation
extra full-graph simulation steps
route and pause bypass
state changes without render acknowledgement
domain replacement without validation
subscriber views that skip direct-API mutations
replay and persistence divergence
non-reproducible smoke or automation results
```

This is primarily an authority and test-reliability defect. The browser global should not be treated as a security boundary, but it must not publish more mutation capability than the product and fixtures intentionally support.

## Required gameplay command flow

```txt
PublicCommandEnvelope
  -> capability lease validation
  -> host generation validation
  -> runtime/run/session admission
  -> lifecycle and route admission
  -> command allowlist
  -> payload schema validation
  -> composite command transaction
  -> committed gameplay result
  -> render-frame acknowledgement
  -> public result publication
```

## Manual stepping rule

Production RAF and external/manual stepping must share one single-writer lease.

```txt
production browser mode
  -> RAF owns step writer
  -> public manual step rejected

fixture mode
  -> RAF disabled or relinquishes lease
  -> fixture capability owns bounded step writer
  -> every step carries expected tick and host generation
  -> result is journaled and rendered or explicitly headless
```

## Required fixtures

```txt
direct resource API unreachable
direct pressure API unreachable
direct orchard collection API unreachable
direct domain command unreachable
direct domain tick unreachable
extra step rejected while RAF owns writer lease
fixture step admitted only under fixture capability
duplicate domain replacement rejected
all admitted gameplay commands produce a typed result and frame receipt
```
